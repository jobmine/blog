文件压缩 (Compression) 之一二

每次打包行李的时候我都觉得要是能压缩东西到无比小的话那简直太幸福了。

Logical compression (逻辑压缩) 利用特定数据的特点进行有逻辑性的压缩（如 png 格式转换成 jpg）。
Physical Compression (物理压缩)，只利用字面上的代码进行压缩（如 css 文件化简）。

Lossy Compression (有损压缩) 会牺牲文件质量进行压缩（如降低图片分辨率）。
Lossless Compression （无损压缩）保留文件原质量进行压缩（如 zip 压缩）。

Character Encoding

字符编码（Character Encodings）是从一个字符 (charset) 到二进制码 (binary string) 的匹配 (matching)。

字符编码方法包括早期的摩斯电码 (Morse Code)，爆豆特(?)码 (Baudot Code)和现在广为采用的 ASCII 和 Unicode。

今天我们来说五种不同的编码：

编码中文 简写 编码英文
游程编码 RLE Run-Length Encoding
霍夫曼编码 Huffman Coding
“搬到前面” MTF Move-to-Front Transform
藍波-立夫-衛曲編碼法 LZW Lempel-Ziv-Welch
BWT Burrows-Wheeler Transform

RLE

RLE 简单说就是一种把加法换成乘法的方法，也就是说：

这种方式的优点显而易见，就是如果有大量重复的加法，我们可以换成乘法，但是如果没有大量重复，就变得事倍功半，比如 1 + 2 + 3 + 4 + 5 + 6 就会被变成 1 * 1 + 2 * 1 + 3 * 1 + 4 * 1 + 5 * 1 + 6 * 1，反而使得压缩后的字符变得更长。

实际的应用：000000000001000011111111111111111111

  • 第一步先写上第一个字符是啥：
    0
  • 然后查有几个，题中有11个0，于是我们算 floor(log(2)(11)) = 3，所以我们写 3 个零：
    0 000
  • 然后，我们把 11 当成二进制写上去：1011
    0 000 1011
  • 然后，我们看下一位，有一位，算 floor(log(2)(1)) = 0：（这里不需要再写在开始的第一位那种，因为0后面分开的一定是1）
    0 000 1011
  • 然后写 1 的二进制，1：
    0 000 1011 1
  • 然后整个写完就是：
    0 000 1011 1 00 11 0000 10100。

讨论：

  • 压缩率最高可以小于 1%（一个极大的重复）
  • 当字符小于 6 的时候，文件不会被压缩，而且
  • 当字符长度为 2 或 4 的时候，长度反而会增加。
  • TIFF 文件使用这种方法。

Huffman Coding

这种方法也叫最优(diao)二叉树，由一个很优(diao)的博士生在MIT发现，当时他的教授说你要么写学期报告，要么来参加期末考试。这个很优(diao)的学生就选了写个更优(diao)的算法好了。于是，他发现并描述了这个方法，并在效率上超过了他导师参与研究的 Shannon-Fano 压缩方法。

因为他解决了影响效率的最大一个问题，从上向下建立树的效率很低。他的解决方案是自底向上形成树。

也就是说，把所有日常（或某个文件中的）字符全部做统计，用比较短的表示出现少的，比较多的表示多的。这个理论有点像英文简写。如果你在一篇文章要出现一个很长的词（比如美利坚合众国马萨诸塞州理科与工科大学电子信息研究学院执行副院长秘书办公室主任王晓明），并且文中会出现20次，那么你会怎样？对，你会写他的简写，（比如美利…..主任王晓明 (下文简写为王主任) ），这样单独20次这个词的压缩率每次就是 3/41=7.3%。

具体的做法是根据字的出现频率，由下而上画出最优二察树。如此 Wikipedia

讨论：

  • 缺点之一是编码真心很辛苦。
  • 没有那只特定的二叉树就没办法解码。（不过比较安全）
  • 非对称性使解码速度很快。
  • (在使用专门的二叉树时，)这货是(目前发现的)最牛x的压缩方法，没有之一。是对字符编码的极限。
  • Huffman 的树不是固定的，针对不同程序的每一个树都不一样，但是他是静止的(static)，也就是说在编码和解码时，用的是完全一样的树。
  • 编码和解码效率相同。

MTF

If we see a character now, we’ll probably see it again soon.

MTF 将这种思想发挥到极致。刚才的方法虽然很好，但是我们需要一个 static 的树跟着我们。可是有没有一种办法可以不用一个树呢。MTF 就是解决办法中的一个。

Iteration Sequence List

b a n a n a a a 1,1 (bacdefghijklmnopqrstuvwxyz)
b a n a n a a a 1,1,13 (abcdefghijklmnopqrstuvwxyz)

b a n a n a a a 1,1,13,1,1 (anbcdefghijklmopqrstuvwxyz)
b a n a n a a a 1,1,13,1,1,1 (nabcdefghijklmopqrstuvwxyz)
b a n a n a a a 1,1,13,1,1,1,0 (anbcdefghijklmopqrstuvwxyz)
b a n a n a a a 1,1,13,1,1,1,0,0 (anbcdefghijklmopqrstuvwxyz)
Final 1,1,13,1,1,1,0,0 (anbcdefghijklmopqrstuvwxyz)

讨论：

  • MTF 是一个适配 (adaptive) 压缩方案。
  • 解码和编码一个速度。

LZW

这个方法就是很厉害的 Huffman，它解决了 Huffman 只能在单独字符重复处理上压缩率很高的弊端，方法是不仅检测单字，同时也检测词组。

具体的方法就是在我们每次取下一个单词之前，确定有没有这个词组。如果有的话就用，没有的话就加到我们的词典里。

具体的方法请参考 如此 Wikipedia

讨论：

  • 当固定 bit 长度时，编码长度是固定的。
  • 字典可能会很大很大很大很大很大很大很大很大，因为每次都在加入新的词组，直到结束。

BWT

这是一个神奇的东西，所以基本的想法就是，根据一种方法来改变原来字符的排序，然后根据某种哲学或者统计学逻辑，认为这样的话会增加字母挨着的几率，然后在进行压缩。

这种方法唯一干的事就是原封不动的把一串东西的顺序改变一下，然后记住顺序，然后通过其他的办法进行压缩（使用 MTF 因为字母重复的时候，MTF 会给一堆 0 which is very nice）。解码的时候呢，先根据编码使用的方法进行解码，然后再用曾经记下的顺序，把这个东西给变回以前的顺序。很好玩，很有效，很不知所云……

具体的方法请参考 如此 Wikipedia

总结

方法 特点
RLE Variable-width, multiple-character encoding
Huffman Variable-width, single-character (optimal in this case)
MTF Adaptive, transforms to smaller integers be followed by variable-width integer encoding
LZW Adaptive, fixed-width, multiple-character encoding Augments dictionary with repeated substrings
BWT Block compression method, must be followed by MTF
